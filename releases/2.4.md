This release delivers a major codebase refactoring that breaks the project into
focused, maintainable modules, along with substantial improvements to the Weather
mode including icon rendering, forecast data, and precipitation display.

### New

- Split `remote_control.yaml` into a thin composition file that imports scoped
  modular packages under `esphome/packages/` — covering actions, button handling,
  display globals, fonts, and UI scripts.
- New ESPHome packages: `remote_actions_*.yaml`, `remote_button_*.yaml`,
  `remote_display_*.yaml`, `remote_fonts.yaml`, and `remote_ui_*.yaml`.
- New C++ modules extracted from the monolithic `entity_helpers.h`: dedicated
  header + source pairs for UI logic, rendering, runtime, sync, input handling,
  bindings, feedback, and state helpers.
- New `include/entity_trackers.h` containing all Home Assistant tracker classes
  that subscribe to and cache entity state independently.
- New `include/ui_state_helpers.h` with persistent UI save/restore packing
  helpers and legacy migration logic.
- New `include/remote_ui_bindings.h` for shared UI binding helpers that
  construct reset/sync state objects without duplicating YAML wiring blocks.
- Weather mode now renders condition icons using Material Symbols Rounded
  glyphs: sunny, clear night (moon), cloudy, rainy, and thunderstorm.
- Partly cloudy rendered as two overlapping icons — a sun or moon behind a
  cloud — matched to the time of day.
- Day/night-aware weather icons: condition display switches between the sun and
  moon symbols based on the `is_night` attribute from Home Assistant.
- Forecast-based today's high and low temperature tracking via the HA `forecast`
  attribute, displayed as `HI X  LO Y` on the weather screen.
- Precipitation probability display in the weather mode footer (`RAIN X%`),
  falling back to humidity if precipitation is unavailable.
- Dedicated `weather_symbols` font (size 9) for rendering compact condition icons
  alongside the temperature readout.


### Improved

- `esphome/local_entities.h` now lives next to `secrets.yaml` for a simpler
  compile and configuration workflow.
- `include/entity_helpers.h` and `include/local_entities.h` are now
  compatibility shims so existing setups that reference the old paths continue to
  work without changes.
- `local_entities-example.h` moved to `esphome/examples/` to align with the
  updated folder layout.
- Weather condition labels are clean and human-readable: `partlycloudy` →
  `PARTLY CLOUDY`, `clear-night` → `CLEAR`, dashes converted to spaces.
- Weather mode shows `SYNCING` until both the condition and temperature have
  been received from Home Assistant, preventing partial or misleading renders.
- Forecast JSON parsing is resilient: tries multiple field name variants
  (`temperature`, `temperature_high`, `native_temperature`, `templow`,
  `precipitation_probability`, `precipitation_chance`, etc.) to support
  diverse HA weather integrations.
- README updated with an Architecture Notes section and revised repo layout
  reflecting the new module structure.


### Internal

- All ESPHome YAML action blocks, button scripts, display globals, fonts, and UI
  scripts extracted into focused packages under `esphome/packages/`.
- C++ UI code split from large monolithic headers into separate files:
  `remote_ui_logic`, `remote_ui_renderer`, `remote_ui_runtime`, `remote_ui_sync`,
  `remote_ui_input_logic`, and `remote_ui_feedback`.
- `WeatherStatusTracker` now subscribes to condition, temperature, humidity, and
  forecast attributes independently for reliable real-time updates.
- Forecast JSON parser handles both bare `JsonArray` payloads and wrapped
  `{"forecast": [...]}` object payloads for compatibility with different HA
  weather platforms.


### Notes

- After upgrading, the preferred location for `local_entities.h` is
  `esphome/local_entities.h`. The old path `include/local_entities.h` is still
  supported via a compatibility shim, so no immediate change is required.
- No changes to `esphome/secrets.yaml`, `esphome/settings.yaml`, board files,
  or entity list format. Existing configurations compile without modification.
- The new modular package layout requires ESPHome 2024.x or later (packages
  with `!include` were already required in 2.x).
